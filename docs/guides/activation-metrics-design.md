# Opt-in Activation Metrics Design

Status: design only. No SDK telemetry is implemented.

## Goal

Understand where new users get stuck in the local AgentGuard path without
weakening the SDK's zero-dependency, local-first promise.

The useful activation question is narrow: can a developer get from install to a
local guard proof without confusion?

## Questions Worth Answering

- Do users reach `agentguard doctor` after install?
- Does `agentguard doctor` complete successfully?
- Do users run `agentguard demo` or `agentguard quickstart` after doctor?
- Which framework quickstart is most often selected?
- Do hosted ingest contracts accept or reject events by SDK version?
- Which documented steps produce errors often enough to fix?

## Explicit Consent Model

No telemetry should run by default.

If implemented later, consent must be explicit at the command boundary:

- one-shot CLI flag, such as `--share-activation-metrics`
- environment opt-in, such as `AGENTGUARD_SHARE_METRICS=1`
- repo-local opt-in in `.agentguard.json`, never generated by default

Every opt-in path should print what will be sent before sending anything.

## Allowed Fields

Only coarse activation metadata is allowed:

- SDK version
- Python major/minor version
- operating system family
- command name, such as `doctor`, `demo`, or `quickstart`
- command result: success or failure
- framework choice for `quickstart`
- anonymous random install/session id generated only after opt-in
- coarse error category, such as `import_error`, `trace_write_failed`, or
  `invalid_framework`

## Forbidden Fields

Never collect:

- trace payloads
- prompts, completions, code, diffs, stack traces, or tool arguments
- API keys, secrets, environment variables, or headers
- filesystem paths, repo names, usernames, hostnames, or IP addresses
- `.agentguard.json` contents
- model inputs, model outputs, token text, or provider request bodies
- dashboard project names or customer identifiers

If a field can identify a person, repo, machine, project, or agent payload, it
does not belong in SDK activation metrics.

## Server-side Metrics First

Prefer metrics that do not require SDK telemetry:

- PyPI downloads by version
- GitHub clone/view/referrer data
- docs page views and outbound clicks
- hosted ingest counters by SDK version and accepted/rejected event kind

These answer adoption questions without instrumenting local-only users.

## Zero-dependency Transport

If implemented later, the SDK must use only the Python standard library. Any
network call must be bounded:

- short timeout
- no retries by default
- failure is silent or logged at debug level
- never blocks or crashes the guarded agent

Metrics transport must be separate from `HttpSink` so trace ingestion and
activation metrics do not become confused.

## Kill Switch

Any future implementation needs a hard off switch:

- `AGENTGUARD_DISABLE_METRICS=1` wins over all opt-ins
- metrics endpoint can reject or disable metrics without affecting SDK behavior
- package behavior stays identical when metrics fail

## Recommended First Implementation, If Approved Later

Start with documentation and server-side measurement only.

If local SDK metrics are still needed, add one opt-in CLI-only experiment for
`agentguard doctor` and `agentguard quickstart`. Do not instrument imports,
guards, tracing, sinks, or normal application runtime.

## Non-goals

- no default telemetry
- no runtime agent instrumentation
- no trace sampling
- no dashboard-required SDK behavior
- no dependency additions
- no product analytics hidden inside local guard execution
